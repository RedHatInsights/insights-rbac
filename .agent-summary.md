# Agent Summary: RHCLOUD-46770

## Big Picture

The parity check system verifies that the data in the Django database matches what's stored in Kessel Inventory. Before this change, the background job only checked workspace parent-child relationships. This left a gap: custom roles grant permissions through tuples like "role:UUID has permission:foo" stored in Kessel, but we had no way to verify those permission tuples actually exist. If they're missing or inconsistent, users might see incorrect access control behavior. This change adds a second check phase to the existing parity job. Now, after checking workspace hierarchies, it also loops through each tenant's custom roles and verifies every permission tuple exists in Kessel. Both checks must pass for a tenant to be considered healthy.

## What Changed

- **New checker class**: Added `CustomRolePermissionChecker` in `rbac/management/inventory_checker/inventory_api_check.py` that follows the same pattern as `RoleBindingInventoryChecker`. It accepts a list of permission tuples, converts them to gRPC CheckRequests, and returns True if all exist in Kessel Inventory, False otherwise.
- **Extended Celery task**: Modified `run_kessel_parity_checks_in_worker()` in `rbac/management/tasks.py` to include custom role permission checks after workspace checks. It queries all custom roles for each tenant, builds permission tuples using `CustomRoleV2._permission_tuple()`, and runs them through the checker.
- **Updated stats tracking**: Added `total_custom_roles_checked` counter to the task's return stats. Each tenant's result now includes `custom_roles_checked`, `custom_role_check_passed`, and a `role_results` array with per-role details (UUID, name, permission count, passed/failed).
- **Combined pass/fail logic**: A tenant now passes only if BOTH workspace hierarchy checks AND custom role permission checks pass. This ensures comprehensive parity validation.
- **Comprehensive tests**: Added 5 tests for the checker class (all pass, some missing, no permissions, single permission, all missing) and 5 integration tests for the task (both pass, workspace fails, role fails, no roles, empty role).

## How It Worked Before

The parity check task only verified workspace parent-child relations. Custom role permission tuples in Kessel were never checked, so inconsistencies could go undetected.

## How It Works Now

After checking workspace hierarchies, the task loops through each tenant's custom roles, builds permission tuples for all permissions on that role, and verifies each tuple exists in Kessel. If any permission tuple is missing, the custom role check fails. If any custom role check fails OR the workspace check fails, the entire tenant check fails. This provides two-layer verification: structural (workspaces) and access control (role permissions).

## Manual Test Instructions

Since this is a background Celery task that requires Kessel Inventory gRPC connectivity, manual testing requires a full deployment environment. The automated tests mock the gRPC calls and cover all scenarios.

However, if you have access to a Stage or Dev environment with Kessel Inventory running:

1. **Set environment variables**:
   ```bash
   export PARITY_CHECK_ENABLED=True
   export PARITY_CHECK_ORG_IDS=<your-test-org-id>
   ```

2. **Trigger the task manually** (in a Django shell or via Celery CLI):
   ```python
   from management.tasks import run_kessel_parity_checks_in_worker
   result = run_kessel_parity_checks_in_worker()
   print(result)
   ```

3. **What success looks like**:
   ```python
   {
       "total_tenants": 1,
       "total_workspace_pairs_checked": N,
       "total_custom_roles_checked": M,
       "passed_tenants": 1,
       "failed_tenants": 0,
       "tenants_not_found": 0,
       "tenants_checked": [
           {
               "org_id": "your-test-org-id",
               "workspace_pairs_checked": N,
               "workspace_check_passed": True,
               "custom_roles_checked": M,
               "custom_role_check_passed": True,
               "role_results": [
                   {
                       "role_uuid": "...",
                       "role_name": "...",
                       "permission_count": X,
                       "passed": True
                   }
               ],
               "passed": True,
               "duration_seconds": 0.123
           }
       ],
       "timing": {"avg_seconds": 0.123, "p95_seconds": 0.123, "p99_seconds": 0.123}
   }
   ```

4. **What failure looks like**:
   - If a custom role's permission tuples are missing in Kessel, `custom_role_check_passed` will be `False`, the specific role in `role_results` will show `passed: False`, and the tenant's overall `passed` will be `False`.
   - Logs will show: `WARNING CustomRole: <uuid> does not have the expected permission relations in inventory.`

5. **Edge cases to test**:
   - Tenant with no custom roles: should pass if workspace check passes
   - Custom role with no permissions: should pass (empty tuple list returns True)
   - Custom role with multiple permissions: all should be checked

6. **Monitoring**:
   - Check Celery Beat logs for scheduled executions (runs daily at midnight if `PARITY_CHECK_ENABLED=True`)
   - Look for `INFO` level logs: `Starting Kessel parity checks for N org(s): [...]`
   - Summary log: `Parity check complete. Checked: X, Passed: Y, Failed: Z, Not Found: A, Total workspace pairs: B, Total custom roles: C`
